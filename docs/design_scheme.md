# 羽毛球比赛视频自动剪辑系统设计方案

## 1. 系统架构设计

系统采用模块化设计，主要分为四个核心模块：视频输入模块、AI分析引擎、剪辑决策模块、输出渲染模块。

### 1.1 模块职责

1.  **视频输入模块 (Video Ingest Module)**
    *   负责读取原始视频流（支持 MP4, MOV 格式）。
    *   进行初步的元数据提取（帧率、分辨率、时长）。
    *   执行视频预处理（降噪、去抖动）。

2.  **AI 分析引擎 (AI Analysis Engine)**
    *   **运动员检测 (Player Detection):** 使用轻量级目标检测模型识别场上运动员位置。
    *   **羽毛球追踪 (Shuttlecock Tracking):** 使用专用小目标追踪模型捕捉羽毛球高速运动轨迹。
    *   **姿态估计 (Pose Estimation):** (可选) 分析运动员动作（扣杀、救球）。
    *   **音频分析 (Audio Analysis):** 识别击球声、欢呼声、鞋底摩擦声。

3.  **剪辑决策模块 (Editing Decision Module)**
    *   **事件识别:** 结合视觉和音频数据，判定发球、死球、回合开始与结束。
    *   **精彩程度评分:** 根据回合持续时间、击球频率、音频高潮点计算片段得分。
    *   **时间轴生成:** 生成包含 `start_time`, `end_time`, `score` 的剪辑清单。

4.  **输出渲染模块 (Output Rendering Module)**
    *   根据剪辑清单提取视频片段。
    *   添加转场效果（淡入淡出）。
    *   (可选) 添加慢动作回放（针对高分扣杀镜头）。
    *   合成最终 1080p 视频。

## 2. 核心技术实现路径

### 2.1 运动员检测 (Player Detection)
*   **模型:** YOLOv8-Nano (或 v10)。
*   **优势:** 推理速度快，适合实时或近实时处理；对人体检测准确率高。
*   **策略:** 仅检测 `person` 类别，通过 ROI (Region of Interest) 过滤掉观众区域，仅保留场地内的运动员。

### 2.2 羽毛球轨迹追踪 (Shuttlecock Tracking)
*   **模型:** TrackNet V2 (基于 CNN 的热图回归模型)。
*   **实现:** 输入连续 3 帧图像，输出羽毛球的坐标。
*   **难点:** 羽毛球体积小、运动模糊。
*   **优化:** 使用卡尔曼滤波 (Kalman Filter) 平滑轨迹，预测丢失帧的位置。

### 2.3 得分事件识别 (Event Recognition)
*   **多模态融合:**
    *   **视觉:** 监测羽毛球是否落地（轨迹高度突变或停止）。
    *   **音频:** 使用 `librosa` 分析音频波形，识别清脆的击球声（高频短时能量）。
*   **逻辑:**
    *   `Rally Start`: 发球动作检测 + 首次击球声。
    *   `Rally End`: 羽毛球落地 or 挂网 or 出界判定 (需场地线检测配合)。

### 2.4 精彩片段提取 (Highlight Extraction)
*   **算法:** 滑动窗口机制。
*   **特征:**
    *   `Rally Duration`: 回合时长（长回合通常更精彩）。
    *   `Hit Frequency`: 击球密度（短时间内多次击球）。
    *   `Audio Peak`: 欢呼声或大声呐喊。
*   **评分公式:** $Score = w_1 \cdot Duration + w_2 \cdot HitCount + w_3 \cdot AudioEnergy$

## 3. 处理流程规范

### 3.1 视频预处理标准化
*   **降采样:** AI 推理时将视频缩小至 640x360 或 960x540 以提升速度，但记录坐标需映射回原图。
*   **去抖动:** 针对固定支架手机可能存在的微小震动，使用 OpenCV 的 `GoodFeaturesToTrack` + 光流法计算单应性矩阵 (Homography) 进行稳像。

### 3.2 音频同步分析
*   提取音频流为 WAV 格式（16kHz单声道）。
*   计算短时能量 (Short-term Energy) 和过零率 (Zero-crossing Rate)。
*   将音频事件时间戳与视频帧对齐。

### 3.3 镜头运动补偿
*   由于是固定机位，主要通过裁切 (Crop) 去除边缘无效区域，聚焦球场核心区域。

## 4. 性能指标要求与硬件配置

### 4.1 硬件配置要求
本系统设计了 **GPU加速模式** (推荐) 和 **无GPU轻量模式** (兼容) 两种运行方式。

| 组件 | **无GPU轻量模式** (最低配置) | **GPU加速模式** (推荐配置) |
| :--- | :--- | :--- |
| **处理器 (CPU)** | Intel i5 (8代+) / Ryzen 5 (6核+) | Intel i7 (10代+) / Ryzen 7 |
| **显卡 (GPU)** | 集成显卡 | **NVIDIA RTX 3060** (6GB+) |
| **内存 (RAM)** | 8GB | 16GB |
| **推理框架** | **ONNX Runtime / OpenVINO** | CUDA 11.8 + TensorRT |
| **处理速度** | 约 0.5~1.0 倍速 (1小时视频需1-2小时) | > 3.0 倍速 (1小时视频需<20分钟) |
| **功能限制** | 仅使用NanoDet/YOLOv8n；降低检测频率(隔帧检测)；羽毛球轨迹追踪精度略有下降 | 全功能开启；高精度模型；实时逐帧检测 |

### 4.2 针对无GPU环境的优化策略
1.  **模型轻量化:** 强制使用 `YOLOv8n` (Nano) 或更轻量的 `NanoDet-Plus`，并将模型导出为 `INT8` 量化版本。
2.  **隔帧检测:** 视频流每 3-5 帧进行一次 AI 推理，中间帧使用 **光流法 (Optical Flow)** 或 **卡尔曼滤波 (Kalman Filter)** 进行目标位置预测，大幅减少计算量。
3.  **分辨率降低:** 推理时输入分辨率降至 480p (640x360)，仅在最终剪辑时使用原片。

### 4.3 复杂场景处理机制 (Robustness)

针对实际拍摄中可能遇到的干扰，系统设计了以下处理逻辑：

1.  **球场遮挡/球飞出屏幕/无法看到落地:**
    *   **逻辑推断 (Logical Inference):** 不单纯依赖“球落地”作为回合结束标志。
    *   **超时判定 (Timeout):** 若羽毛球在画面中丢失超过 `2.0秒` 且未检测到击球音，判定为死球。
    *   **行为分析:** 监测运动员动作——若双方运动员停止跑动、直立行走、或有捡球动作，立即标记回合结束。

2.  **远端场地/背景干扰:**
    *   **ROI 区域过滤 (Region of Interest):** 系统启动时（或首帧）进行场地边缘检测，划定“比赛有效区域”。所有在此区域外的运动目标（远端球员、观众）一律过滤。
    *   **主球员锁定:** 初始化时锁定画面中心或最近的 2-4 名运动员，利用 ReID (重识别) 特征持续追踪，忽略背景中穿梭的其他人员。

3.  **路人短暂遮挡镜头:**
    *   **全图异常检测:** 计算帧间像素变化率 (Pixel Difference)。若画面大面积（>60%）发生剧烈变化或变黑/变白，判定为“遮挡事件”。
    *   **自动暂停与恢复:** 遮挡发生时，暂停当前的状态判定（不强制结束回合）。遮挡结束后（画面恢复稳定），立即触发 **全局重检测 (Global Re-detection)** 找回运动员位置。若遮挡时间过长（>5秒），则强制切断当前片段。
